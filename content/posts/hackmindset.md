面對聰明的對手需要與面對自然發生或意外問題不同的心態、方法與紀律。無論我們是在模擬攻擊還是防禦攻擊，我們都需要考慮對手的觀點和可能的行動，並試著預測他們可能的行動。因為我們的對手是有自主性的人類，他們能夠推理、預測， 判斷、分析、推測與深思熟慮。他們也能感覺到 像是快樂、悲傷、貪婪、恐懼、勝利和罪惡感。 攻擊者和防守者都能利用他們人類的情感 對手。例如，攻擊者可能會利用尷尬來判斷 挾持電腦系統並威脅公開其資料。 防守方則可能利用恐懼來嚇退攻擊者 阻止他們進入他們的網絡。這個現實意味著人類是 這是資安的關鍵組成部分。


另一個重要的安全面向是，它通常涉及 在不確定性下推理 。雖然我們有很多演繹能力 技能，我們絕非心智全知。我們無法判斷 從某個真理推論出的一切 ，而我們無法知道或記得無限多的事實。



想想像西洋棋這樣的遊戲和撲克這類遊戲有何不同。在西洋棋中，你知道對手對棋局狀態的一切反應（反之亦然）。你可能不知道他們在想什麼，但你可以根據他們用來判斷的資訊，對他們的下一步做出預測。然而，玩撲克時你無法掌握對手擁有的所有資訊，因此必須基於不完整的數據來做出預測。

從攻擊者與防守者的心理角度來看，資訊安全更接近撲克而非西洋棋。舉例來說，當我們模擬攻擊時，永遠無法知道我們所攻擊的機器/系統/網路/組織的所有資訊。因此，我們必須做出假設並估計機率，有時是隱含的，有時是明確的。

相反地，作為防禦者，我們也無法察覺每一個潛在的攻擊途徑或漏洞。因此，我們需要對沖風險，確保最有可能受攻擊的攻擊面獲得充分保護。

智慧對手的問題與不確定性的問題都顯示，理解網路安全需要更多了解我們作為人類代理人的思考方式，以及如何思考 解決問題。這表示我們需要專門收養和培育 這些心態將幫助我們學習並應用技能。

### 關於心態的一句話

安全不僅是理解技術與程式碼，更是理解你自己與對手的思維。我們往往把心態看作一套信念， 用來塑造我們對某件事的個人觀點。

兩個鮮明的心態例子是固定的 心態和成長心態。

一個心態固定的人相信自己的技能、天賦或學習能力就是這樣，努力提升也無法獲得任何收益。

另一方面，成長型心態鼓勵相信心理能力是靈活且可適應的，且個人能隨時間提升學習能力。

研究顯示，例如，我們相信自己能夠從錯誤中恢復的心態 這讓我們在這方面有明顯的進步。這只是其中一個面向 成長型思維，但這很重要。這是因為安全性 要求我們犯錯並從中學習，並且持續保持 學習與重新評估。

另一個極具價值的心態是恰如其分的說法 安全心態 。由資安研究員 Bruce Schneier 提出。這種心態鼓勵人們不斷質疑如何攻擊（或防禦）一個系統。如果我們能在遇到新穎的想法、機器、系統、網路或物件時自動提出這個問題，就能開始注意到各種反覆出現的模式。


在 OffSec，我們鼓勵學習者採用 「努力嘗試」的理念 心態。為了更好地理解這種心態，讓我們快速考慮兩個 在「失敗」時刻的潛在觀點。

如果我的攻擊或防禦失敗，這既代表了我目前技能/流程/配置/方法的真實，也反映了系統的真相。

如果我的攻防失敗，這讓我能學到新東西、改變方法，並做出不同的嘗試。

這兩種觀點幫助人們具備心理韌性，讓他們能犯錯並從中學習，這在任何資安子領域中都至關重要。關於如何學習及「努力一點」心態的更多資訊，可參考本入門學習路徑中的「有效學習策略」模組。

模仿對手的思維

值得停下來思考 我們將特別關注 進攻 即使在我們許多防禦課程和模組中，也包含在安全性方面。 有人可能會好奇，為什麼一位資安專業人士會是 利益與目標是捍衛一個網絡、組織或政府 也應該學習進攻端。

我們來比喻一位中世紀君主建造城堡。如果君主得知敵人有能投擲巨石的投石機，他們可能會設計城堡牆壁更厚。同樣地，如果敵人裝備有梯子，君主可能會給部隊工具，讓他們把梯子推下城牆。


這位君主越了解潛在攻擊者，越能像攻擊者一樣思考 ，防禦就越好 造。君主可能會從事「攻擊性」活動，或 審計以了解自身防禦的漏洞。例如，他們可以進行「戰爭遊戲」，指揮自己的士兵互相模擬戰鬥，幫助他們充分理解真實攻擊者的能力與破壞力。

在資安領域，企業可能會聘請個人或公司執行滲透測試，也稱為滲透測試 。滲透測試員扮演攻擊者的角色，以更深入了解系統的漏洞與暴露的弱點。


善用攻擊者的技能與心態，讓我們能更好地回答以下問題：

「攻擊者怎麼可能進入？」


「他們能用這個權限做什麼？」

「攻擊最壞的結果是什麼？」

雖然學習駭客技巧對有志成為滲透測試者來說當然是必不可少的，但我們也相信防禦者、系統管理員和開發者至少能從基礎上學習攻擊性技術與技術中獲益良多。

相反地，我們的經驗是，許多最優秀的滲透測試員和網頁應用程式駭客，往往是那些有廣泛網路防禦、建置網頁應用程式或系統管理經驗的人。


在這個故事裡，讓我們想像鮑伯有一項他想要守護的資產：一棵偉大的香蕉樹！鮑勃想確保只有他能摘香蕉。同時，攻擊者愛麗絲最想做的就是偷走鮑勃的香蕉。

首先，Bob 並不特別注意他樹木的安全。愛麗絲走過去偷香蕉相對容易。隨著愛麗絲越來越擅長偷竊，鮑勃也會越來越擅長保護他的樹。

當鮑勃第一次意識到愛麗絲的背叛時，他得知守衛能阻止愛麗絲偷香蕉。但愛麗絲推測鮑伯總有一天會睡覺。她注意鮑伯何時入睡，然後悄悄溜上樹偷竊。

鮑勃隨後想出如何在樹周圍築起高大的石牆。愛麗絲努力突破或攀爬。最終，她學會了如何挖牆底。

接著，鮑勃訓練了一隻守衛犬來保護這棵樹。愛麗絲發現她可以用零食安撫狗狗。

鮑勃修習硬體安全課程，安裝攝影機和警報器，隨時提醒愛麗絲在附近。愛麗絲學會了如何關閉監視器和警報器。

這個循環幾乎可以無限期地持續下去。奇怪的是，攻擊方和防守方彼此依賴來提升技能組合，並更深入理解各自的技藝。




But we how to learn ???


想到教育可能會讓人感到有些不知所措，所以我們先試著用更簡單的方式來理解它。我們能證明自己「學到了」某件事的方法之一，就是能夠創造並取回一段記憶。

例如，我們可能會學習一個特定的指令，在 Linux 中重新命名檔案，mv oldfilename.txt newfilename.txt。之後，我們可能會發現自己坐在電腦前，需要重新命名檔案。我們希望在這種情況下，遠離教科書和任何教學資料，能記住這個指令和語法。理想狀況下，我們可以從記憶體輸入指令並成功重命名檔案。

大量研究探討記憶如何運作 ，以及我們如何創造深刻的記憶並學習新技能。全面回顧所有細節不在本模組的範圍，但簡而言之，我們可以透過以下步驟來提升記憶力：


提升我們接收資訊的品質



改善我們接收資訊的方式或方式



改進我們檢索資訊的做法


我們在這三個方面越努力改進，記憶和學習的能力就會越好。我們也知道，在切換傳遞方式時重複資訊也很有幫助。


透過第二種方式吸收相同資訊，例如先閱讀說明再觀看同一模組的影片，稱為雙重編碼 。雙重編碼的基本原理是，透過不同方式反覆學習相同資訊，能提升記憶力。


![alt text](/image.png)

上方所示的圖片不僅僅是雙重編碼的說明;它是雙重編碼本身的例子。透過結合解釋閱讀概念過程的文字段落，並搭配說明性視覺輔助工具，資訊能更好地烙印在我們的大腦中。

越來越多的研究，包括可重複的實驗和神經影像的證據，支持雙重編碼作為有效的學習策略 。

在豪爾赫·路易斯·博爾赫斯的虛構故事中，有個名叫富內斯·紀念者的角色能生動地記得他目睹的每一件事情。可惜的是，我們大多數人並沒有這樣的天賦。我們在學習（或創造記憶）時，最常遇到的兩個問題是「太久以前」或「一次資訊太多」。

讓我們先從遺忘的問題開始。1885 年，博學科學家赫爾曼·埃賓豪斯開始記憶幾份文件，並反覆測試記憶中的內容。他只有在背完後立刻測試自己，才能記住所有細節。艾賓浩斯發現，他在取得時只能記得百分之百的資訊。之後，他開始很快忘記資訊。等了 20 分鐘，他只記得 58%。一天後，他只記得 23%。他稱這種衰退為「 遺忘曲線 」。

第二個問題，我們稱之為「一次資訊過多」，通常稱為認知負荷 。


為了更好地理解認知負荷，可以把我們的大腦想像成一個房間，裡面有佔據空間的資訊片段不斷進出。到了某個階段，如果越來越多資訊不斷湧入，空間就無法讓所有東西保持有序。很快，房間就擠滿了，沒有空間讓更多人從門口進來。


為了解決這個問題，教練可能會嘗試減少所謂的「額外負擔」。這些都是額外的新資訊，並不重要或必要。


面對困難

有句常見的說法是「熟能生巧」。這或許是真的，但這也引出了一個問題，什麼才是理想的練習？

讓我們來考慮以下幾點 該實驗於 1978 年進行。一組 8 歲的孩子被分成兩組，練習一個簡單的任務：把一個小豆袋丟進目標洞裡。在接觸目標距離三英尺（約 90 公分）後，各組花了三個月時間練習。有一組則持續練習，目標保持在同一距離。另一位則瞄準兩個目標——練習距離分別為兩英尺（60 公分）和四英尺（120 公分）。

最後的測試任務是將豆袋丟向三英尺外的目標。那群花了整天在那個距離練習的隊伍，被練習在兩英尺和四英尺的隊伍擊敗。


這些及其他研究顯示，掙扎不僅對學習經驗重要，更重要的是建立幫助我們學習新技能的神經通路。



這種必須奮鬥的過程，意味著我們在 OffSec 的學習教材中不會做太多精確的重複。由於學習是自我導向的，尋求更多重複的學習者可以無限次回到特定部分。


我們常常選擇走一條間接的路線，而不是重複這種重複。例如，我們可能會嘗試一些無效的方法，好讓自己能重新振作起來再試一次。這比喻上就像稍微移動目標。


 戰術與常用方法

接著，我們需要思考策略與戰術。請參考孫子的以下引言：

沒有戰術的策略是通往勝利最慢的路。沒有策略的戰術是失敗前的噪音。

從最基本的意義上來說，我們可以把策略看作是一種長遠的願景，而戰術則是我們採取的短期、即時行動。策略是地圖，戰術是步驟。

康乃爾註釋

有許多不同的筆記系統。讓我們簡單看看一個叫做康乃爾筆記的工具，這是康乃爾大學教授沃爾特·保克（Walter Pauk）在 1950 年代開發的。此方法使用筆和紙，有助於雙重編碼。

第一步是將頁面分成三個區域。這些就是提示 （頁面左側）、 註解 （頁面右側的大區域）以及摘要 （頁面底部幾行空白）。


提示可能是我們對文本、關鍵字或片語的問題。舉例來說明，我們來討論一個模組，例如密碼雜湊。這個模組可能包含關鍵術語，例如單向加密 、 加鹽和破解密碼 。我們也可能有問題，例如：「有些雜湊方法比其他方法更好嗎？」


## 章節一：主要內容

我們常常以為自己知道的已經足夠——在滲透測試或漏洞研究裡，這種自信會讓你忽略真正重要的線索。亞當．格蘭特在《逆思維》裡說到的三種錯誤心態：**傳教士（Preacher）**、**檢察官（Prosecutor）**、**政治人物（Politician）**，在駭客圈同樣常見。下面我把每一種心態拆開講，並附上實例，說明在安全工作中它們如何讓你錯過真相——以及如何像科學家一樣避免。

### 傳教士心態（Preacher）：我相信的就是對的  
我們常常把初步的發現當作信仰。一旦發現了一條假設（例如：這個 web 應用的某個參數可能導致 SQL injection），我們就會緊抱這個假設，努力論證它，而忽略新的反例或邏輯漏洞。

**駭客實例：**  
你在一個 Web 應用上測試到 `id` 參數有反應（回傳不同錯誤或延遲），你的第一直覺是「這是 SQLi」。你開始在 payload 上投入更多時間，調整引號、盲注技巧、payload 編碼，並用工具生成大量請求去證明 SQLi 存在。結果花了好幾小時卻沒有成功。後來你回頭檢查日誌，發現那個 `id` 參數其實會觸發一個後端 cache miss，導致延遲；真正問題是未授權的 API 端點返回不同格式，並非 SQL 注入。

**教訓與改進：**  
像科學家一樣，把你的發現當作假設 `H0: 這是 SQLi`，去設計**反事實測試**（例如：傳送非數字、傳送長字串但無引號、對該參數做 HEAD 請求觀察差異）。如果反證明了你的假設，就接受並改變方向，而不是硬扛。

---

### 檢察官心態（Prosecutor）：我在找證據把你定罪  
我們在看到別人或工具提出一個假設時，會像檢察官一樣只找支持我們觀點的證據，目的是證明對方錯誤或證明自己正確。這會導致 confirmation bias（確認偏誤）。

**駭客實例：**  
在紅隊演練中，先前某次偵測告警提示可能存在 RCE（Remote Code Execution）。你熱衷於證明這個 RCE 是可利用的，於是只跑既有的 exploit 模組並忽略環境限制（比如 Web 應用在容器內運行，沒有寫入權限）。你找到一些支持點（例如能導致 command 回顯的 payload），但每次都是不穩定或依賴特定 header。你在內部報告裡把不穩定現象包裝成“RCE（需要某些特殊情況）”，讓業務誤以為系統可以被穩定攻破。

**教訓與改進：**  
檢察官心態會讓報告誇大風險。科學家會記下條件與約束：在報告上標明「在以下條件下可以觸發（不可在預設情況下重現）」，或列出不可重現的假陽性。安排可重現性測試（相同環境、相同步驟）來驗證穩定性，避免過早下定論。

---

### 政治人物心態（Politician）：為了支持、為了群眾而說話  
在團隊或社群中，當你想要取得信任或資源（例如要上線 exploit proof-of-concept），你可能會調整說詞以討好聽眾，或只呈現最容易被接受的結果。

**駭客實例：**  
你想推動一個大規模的安全改進專案（例如替整個微服務導入集中式身份驗證），管理層要求風險評估報告。為了讓專案更容易被批准，你在報告中淡化成本、誇大小幅度成功率，或忽略需要停機測試的真實風險。專案批下來後才發現實際改造範圍與風險超出預估，導致回滾或人力耗損。

**教訓與改進：**  
科學家會把事實與期望分開，並展示多個情境（最佳/常態/最壞）。對管理者先說清楚「假設條件」與「不確定性」，這比討好式的承諾更能長期建立信任。採用 A/B 試點而非一次性大改，可以降低政治壓力帶來的錯誤決策。

---

### 科學家心態（Scientist）：把想法當成假設，擁抱反證  
理想的心態是把每個觀點視為一個待驗證的假設，用實驗去證明或反駁，而不是保護想法。

**駭客實例：**  
你懷疑某廠商使用的第三方套件存在未被揭露的 deserialization 漏洞。科學家做法是：建立一個最小可重現環境（container）、編寫可控 payload 測試序列化/反序列化流程、做 instrument（比如在反序列化入口打點），並設立對照組（不載入該套件或載入安全版本）。你會記錄所有實驗步驟、變數、成功率，並在發現可利用鏈時一層層拆解依賴，而不是直接貼上「高風險」標籤。

**教訓與改進：**  
科學家做法讓你的發現具有可重現性與說服力，也能減少 false positive。在報告中包含實驗環境（OS、lib 版本）、payload 與測試 log，讓修補方能重現並優雅修補。

---

### 初始假設到反覆驗證的流程（How-to in practice）  
我們在 hacking 的每一步都應該問三個問題並實際操作它們：

1. **那是「真相（What is the truth）」嗎？**  
   - 實例：你以為 `X-Forwarded-For` header 可繞過 IP 限制。先把 header 設為幾種值（私網、空值、偽造），觀察後端真實日誌（如果可得）或用 proxy capture 檢查後端回應，確認不是只是前端行為差異。

2. **為什麼是 / 為什麼不是？（Why yes / Why no）**  
   - 實例：若發現 NTLM relay 有機會，問「環境是否啟用了 SMB signing？域控是否已經 patched？」這些會直接決定技術可行性。不要停留在單一測試結果上，逐項排查假設。

3. **如果不是，下一步是什麼？（If no, then what）**  
   - 實例：若 RCE 不穩定，下一步不是一味嘗試 payload，而是改成做 instrumentation（增加 log、trace）、回顧部署環境或與開發討論異常條件。也可能改成攻擊較弱的 side-channel（例如 SSRF → 配置誤用）。

4. **如果是，是否該再雙重檢驗（Double think）？**  
   - 實例：你拿到 root shell，先不要得意忘形。先做「最小動作驗證」：列出環境、抓取限制性檔案（/etc/hosts、/proc/version）、檢查是否有 sandbox 或容器化限制，確保這個權限能夠在實際情境中維持而非短暫可被隔離。

---

### 達克效應（Dunning–Kruger）與冒充者症候群（Impostor Syndrome）——在駭客圈的兩端  
我們會看到兩種極端：一端是沒有足夠技能卻過度自信（達克效應），另一端是能力很強卻不相信自己（冒充者症候群）。兩者都會阻礙學習與決策。

**駭客實例（達克效應）：**  
一個剛學會基本 Nmap 與 Metasploit 的新人，看到「某某漏洞一鍵利用」，就聲稱「整個系統都不安全」，於是發表誇張的漏洞通告，結果被資安社群指出環境設定是關鍵（比如防火牆在外側），他的結論被削弱，信譽受損。

**駭客實例（冒充者症候群）：**  
一位資深紅隊成員在內部會議中總是不敢直接發表高風險假設，擔心自己說錯會被笑話，結果錯失在早期告知開發團隊一個可能的權限邊界問題的機會，直到被攻擊演練揭露才追悔莫及。

**教訓：**  
- 對抗達克效應：靠可重現的實驗、peer review、寫 test-cases、讓發現經得起他人驗證。  
- 對抗冒充者：鼓勵分享小失敗（postmortem）、建立心理安全的回饋機制，讓人人知道「沒有人永遠全對」。

---

### 自信的謙遜（Confident Humility）——駭客應有的平衡姿態  
最理想的狀態是「相信自己能學習和解決問題，但對自己的知識持謙虛態度」。這在安全領域尤其重要：技術快速更新，今天的 exploit 明天可能就被 patch。

**駭客實例：**  
你在 exploit 開發過程中，某個技術細節你掌握得很好（比如 heap feng shui），但你仍然承認在某些平臺上不熟悉（如 ARM vs x86 的差異）。所以你主動去找擅長 ARM 的同事做 code review，並在手冊裡標注兼容性限制，而不是把自己當作全能專家——這樣能加快修正與防護速度。

---

### 享受犯錯（Enjoy Failing）——錯誤就是快速學習的燃料  
我們很多時候把錯誤視為恥辱，但在研究與 exploit 開發中，錯誤其實是最有價值的訊號。每一次失敗都指出哪裡被我們誤解了。

**駭客實例：**  
你嘗試對一個文件解析器做 fuzzing，連續數百次 crash 都沒有得到有用的輸出。你改變策略：把 fuzz 策略細分為幾個 module 並添加更多崩潰上下文（stack trace、heap dump）。雖然最初連輸出都亂七八糟，但通過整理崩潰模式，你最終定位到一個邊界檢查錯誤，發現真正的利用點。你在 Internal Postmortem 裡把每次失敗的訊息記下來，下一位工程師能更快重建你走過的路。

---

### 分離「觀點」與「身份」——別把漏洞當成個人攻擊  
我們很容易把某個想法視為「我就是這想法」，當有人反駁時會感到被冒犯。若把觀點視為暫時性、可替換的工具，就能更冷靜地處理質疑。

**駭客實例：**  
在 code review 中你堅持某段防禦邏輯（CWE 修補方式），另一個同事提出不同做法。若你把自己綁定於「我就是那個做法」，會馬上情緒化反駁。相反，把「這只是一個假設」放在桌上，雙方用測試案例驗證哪個做法在更多情境下是安全或效率更佳，最後選擇資料支持的方案而非情緒勝負。

---

### 任務衝突 vs 人際衝突：鼓勵健康的「任務衝突」  
團隊裡若只講和氣、避免衝突，表面上和諧但會隱藏大量問題。高效團隊要能產生**任務衝突**（針對想法的爭辯），但避免人身攻擊。

**駭客實例：**  
在藍隊/紅隊回顧會（post-exercise review），若團隊只說「做得很好」，就失去學習契機。良好做法是指定「守門人」角色負責挑戰每個假設（例如：這個 detection 規則在什麼情況會漏掉？若 attacker 用多跳 C2，該規則還能 detect 嗎？）。這類任務衝突會帶來更強的 SOC 規則，而不是只會讓人互相抱怨。

---

### 能說服別人的逆思維方法（從檢察官到引導者）  
當你試圖說服別人（例如推動修補、說服管理層投資安全），不要直接轟炸事實與數據。更有效的是先建立共同點、聚焦最強論點、並用提問引導對方自己得出結論。

**駭客實例：**  
要說服產品經理重視一個低頻但高影響的供應鏈風險（例如 CI/CD pipeline 的憑證洩露風險），不要一開始就放一長串漏洞清單。先承認「我們都希望快速部署、滿足用戶」，建立共同目標；接著挑兩個最致命的情境（例如：憑證洩露→攻擊者推惡意鏡像→全站被植入），用故事式案例說明實際影響；最後提問：「如果發生這種情況，現在我們能在多久內回復？」讓對方思考回復成本，自己推導出需要投入的改善措施。

---

### 動機式會談（Motivational Interviewing）在資安溝通中的應用  
改變不是靠命令，而是靠幫對方發現改變的內在動機。這種技巧用在疫苗溝通非常有效，在安全教育與推動改變上也同樣適用。

**駭客實例：**  
在推動開發團隊使用安全庫（比如使用更安全的加密套件）時，不要只講「這是安全的」。先聽開發的痛點：「我們怕性能問題、整合成本太高」。用開放式問題引導：「如果我們能在不影響性能下，自動化套件升級，你會願意嘗試嗎？」用同理心承認開發考量，然後用小步驟（pilot）降低阻力，讓改變變成開發者自己的選擇而非強制指令。

---

### 在處理高度爭議議題時呈現「複雜性」能降低極端化  
當問題被極端化成黑白二分時，人們更難妥協。展示問題的灰色地帶能促成更理性的討論。

**駭客實例：**  
在公司內部針對「是否公開漏洞細節」出現激烈對立時，單純說「絕對公開」或「完全不公開」會造成對立。相反，把情境拆成多個維度（影響範圍、已存在的 mitigations、是否有人正在被利用、是否存在 patch）並用案例展示複雜性（例如某次公開導致被利用但同時加速修補），能促使雙方找到折衷（例如有條件的公開或延遲披露）。

---

### 在教育與訓練上，教學生/同事「如何思考」更勝於教他們「要思考什麼」  
講座式教學常把學生變成被動接受者，而模擬實驗、互評、迭代設計能培養他們檢驗自己假設的能力。

**駭客實例：**  
在內部安全訓練中，如果只教「怎麼用 Burp 進行 XSS 測試」，學員掌握工具卻不知道為何這些 payload 有效。更好的方式是給一個不完備的範例應用，讓學員找出錯誤（過時的輸入過濾、錯誤的 output encoding），彼此評審後重寫過濾邏輯。當他們自己發現問題並修正，學習效果遠勝於只是看 demo。

## 章節二：心得與反思（敘述 + 實務應用）

讀完《逆思維》後，我更加確信：**駭客的核心不是炫技，而是保持好奇與懷疑**。在實務上這帶來幾個改變：

- 我會在每個假設旁邊寫上「可被證偽的測試」，確保研究不是單向論證。  
- 我會在紅隊/藍隊報告中加入「重現步驟」與「限制條件」，避免誤導管理層。  
- 我會在團隊文化裡推動小規模的失敗分享會（failed experiments），把錯誤視為資產而非恥辱。  
- 我會把溝通方式從「我說服你」改為「我引導你發現」，特別是在要改變流程或工具時。
